# 前言

AI Factory 值得系统讲解，是因为它把过去相对分散的应用平台、模型训练、推理服务、GPU 集群、网络存储、机房工程和财务模型连成了一条生产链。过去的数据中心主要生产通用计算能力，AI Factory 则持续生产模型能力与 token，并且每一次输出都可以被延迟、吞吐、成本、质量和可靠性度量。

传统云计算知识仍然是底座，但它无法单独解释大模型时代的基础设施约束。Kubernetes 的调度语义很重要，却不能自动解决 gang scheduling、GPU 拓扑、NCCL 通信和长时间训练任务的稳定性。虚拟机和裸金属交付很重要，却不能自动说明 KV Cache 如何改变显存压力，或 prefill/decode 如何影响 TTFT 和 TPOT。对象存储、文件系统和网络架构也很重要，但在训练场景里，它们必须和 checkpoint、数据流水线、AllReduce、RDMA、故障域一起分析。

也不能只从 GPU 或模型看问题。只看 GPU，容易把 AI Factory 误解成硬件采购和运维；只看模型，容易忽略请求治理、作业调度、准入验收、容量规划和成本结构。本书采用“从应用到 GPU 基础设施”的路径，是为了让读者看到需求如何逐层向下传导：应用体验决定 token 模式，token 模式决定推理服务，推理服务决定 Runtime 和调度策略，调度策略决定 GPU IaaS、网络、存储和物理设施的设计。

本书会区分三个容易混用的概念。AI Factory 是完整生产系统，覆盖从应用到基础设施的端到端能力。Token Factory 是观察 AI Factory 产出的经济性视角，关注 tokens/s、tokens/W、cost per token、revenue per token 等指标。TokenFoundry 不作为通用技术名词使用，它更适合在产业案例中讨论某类组织如何围绕 token 生产和商业化组织资源。

本书面向有云计算、Kubernetes、后端工程基础的读者。你不需要已经是 CUDA、NCCL 或数据中心电力专家，但需要愿意把应用、平台、模型、运行时、调度、硬件和经济性放在同一个系统里思考。
