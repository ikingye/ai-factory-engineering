# 写作规范

本书的目标不是收集 AI Infra 名词，而是把 AI Factory 当作一个端到端生产系统讲清楚。每一章都应帮助读者回答三个问题：这个问题为什么存在，它在系统中如何被实现，生产环境中如何观察、验收和诊断。

## 章节质量标准

一章成熟正文应至少包含以下要素：

- 一个真实工程场景：从故障、需求或上线压力切入，而不是从定义切入。
- 一组核心概念：首次出现的术语给出中文解释和英文原词。
- 一张系统关系图：能用 Mermaid 表达组件关系、数据流或控制流时，优先画图。
- 一条工程路径：说明请求、任务、数据、模型或资源如何穿过系统。
- 一组常见故障：列出真实生产中会遇到的错配、瓶颈和排障入口。
- 一组指标：区分体验指标、资源指标、稳定性指标和成本指标。
- 一组设计取舍：解释为什么不是只有一个正确方案。
- 小结：用 3-5 条 bullet 收束本章的工程判断。
- 延伸阅读：优先引用官方文档、经典论文和可公开验证的工程案例。
- 跨章节覆盖：复杂主题可以拆到多个章节，但必须在全书范围内形成闭环，不能让读者读完仍缺关键机制。

## 编辑工作流

本书后续扩写采用 `doc-coauthoring` 风格的三段式工作流。该 skill 来自 Anthropic `skills/doc-coauthoring`，核心思想是先补齐作者上下文，再逐节精修，最后用读者问题测试文档是否真正可用。本书不把这个流程当作形式化步骤，而是把它转化为每章的审稿门槛。

第一步是上下文收集。扩写一章前，先明确文档类型、目标读者、读完后的预期能力、已有模板、相关章节和外部证据。对于本书，默认读者是具备云计算、Kubernetes、后端工程基础，但尚未系统理解 AI Infra 的工程师；成熟章节应让读者能解释机制、落地实现、设计取舍和排障路径。

第二步是逐节精修。不要一次性重写整章，而是按场景、概念、架构、技术机制、工程实现、故障、指标、取舍逐节推进。每一节都要先问“读者需要拿走什么判断”，再补机制、路径、配置、指标和反例。若一段文字只是在重复标题、罗列名词或表达正确但不可操作，应删除或改写。

第三步是读者测试。完成一章或一条主题链路后，要模拟一个新读者提出 5-10 个真实问题，例如“GPU 空闲但 Pod 为什么 Pending”“容器里 `nvidia-smi` 正常为什么 NCCL 仍然失败”“一次 Gateway fallback 如何影响计量”。如果本章或跨章节链路不能让读者回答这些问题，就继续补上下文、图、配置或故障树。

每轮扩写完成后，应检查：

- 读者是否能从本章定位它在 AI Factory 分层中的边界。
- 读者是否能说出至少一条端到端路径或状态机。
- 读者是否能把概念映射到配置、事件、日志、指标或验收动作。
- 读者是否能区分相邻系统的职责，而不是把 Gateway、Scheduler、Runtime、IaaS 和物理资源混在一起。
- 读者是否能回答本章最常见的生产故障如何诊断。

## 写作口径

- 使用中文叙述，关键技术名词保留英文。
- 不把 AI Factory 等同于 GPU 集群。
- 不把 Token Factory 等同于 AI Factory；它是产出度量和经济性视角。
- 不把 Kubernetes、Slurm、容器和 GPU 调度简单归为 PaaS；它们属于资源编排与作业调度层。
- 对快速变化的实现使用中性表达，例如“常见做法是”“一种典型架构是”。
- 不虚构公司私有数据，不编造性能数值，不把示例值写成行业结论。
- 引用具体产品能力时优先查官方文档；不确定时明确说明边界。
- 不用字数掩盖深度不足。允许章节变长，但增长应来自机制、路径、图表、配置、故障、指标和验收内容。

## 章节模板

后续扩写章节时建议使用以下结构，并允许根据主题增删小节：

```markdown
# 第 X 章：章节标题

## X.1 本章回答的问题

## X.2 本章上下文

## X.3 读者测试

## X.4 一个真实场景

## X.5 核心概念

## X.6 系统架构

## X.7 关键技术

## X.8 工程实现

## X.9 常见故障

## X.10 性能指标

## X.11 设计取舍

## X.12 小结

## X.13 延伸阅读
```

## 标题编号规范

章节标题使用 `# 第 X 章：章节标题`。所有二级标题都必须连续编号，格式为 `## X.N 标题`，其中 `X` 是章号，`N` 是本章内从 1 开始递增的顺序号。

导读、上下文、读者测试、场景、概念、架构、技术机制、工程实现、故障、指标、取舍、小结和延伸阅读都参与编号。例如第 21 章应从 `## 21.1 本章回答的问题` 开始，后续依次是 `## 21.2 本章上下文`、`## 21.3 读者测试`，直到 `## 21.22 延伸阅读`。这样目录、正文引用、审稿意见和外部讨论都使用同一套位置坐标。

编号必须连续，不能因为某一节是模板节就跳过。新增或删除 H2 后，应重新排列本章编号，避免出现 `21.6` 后直接跳到 `21.10`，也避免一个章节内同时存在编号和未编号二级标题。第 0 章使用 `0.1`、`0.2` 这类编号。

扩写章节后运行：

```bash
python3 tools/audit_doc_coauthoring.py
python3 tools/audit_heading_numbering.py
```

`tools/audit_heading_numbering.py` 会检查每章所有 H2 是否都带有当前章号前缀，以及编号是否从 1 开始连续递增。编号的意图不是制造形式负担，而是让章节目录像经典技术书一样可引用、可审校、可长期维护。

## 图表规范

Mermaid 图应服务于论证，不做装饰。推荐使用三类图：

- `flowchart`：表达请求路径、训练链路、控制面和数据面关系。
- `sequenceDiagram`：表达一次 API 调用、流式返回、多轮 Agent 调用。
- `graph` 或 `flowchart TB`：表达分层架构、依赖关系和故障域。

图中节点名称应短，正文负责解释细节。不要把整段文字塞进图里。

## 指标规范

指标应说明口径和用途。至少区分：

- 用户体验：TTFT、TPOT、端到端延迟、错误率。
- 平台治理：QPS、RPM、TPM、限流命中、队列等待。
- 资源效率：GPU 利用率、HBM 占用、KV Cache 使用、tokens/s。
- 稳定性：重试率、超时率、NCCL 错误、节点健康。
- 经济性：cost per token、revenue per token、tokens/W、毛利。

## 延伸阅读规范

延伸阅读优先级：

1. 官方文档。
2. 经典论文。
3. CNCF、Kubernetes、NVIDIA、OpenTelemetry 等公开工程资料。
4. 可验证的工程案例。

如果暂时没有把握，不要编造链接；可以先引用更稳定的官方入口页，并在后续深度审校中替换为更精确的章节或规范链接。

## 深度化要求

后续更新应参考 `docs/depth-standard.md`。一个主题不要求全部放进单章，但全书合起来应覆盖它的核心知识点。例如 GPU 容器主题应在 GPU 软件栈、容器与 Kubernetes、GPU on Kubernetes、镜像驱动初始化、准入测试等章节形成闭环。

发布前必须同时运行 `python3 tools/audit_doc_coauthoring.py`、`python3 tools/audit_heading_numbering.py`、`python3 tools/audit_depth.py --limit 160`、`python3 tools/audit_placeholders.py` 和 `mkdocs build --strict`。
